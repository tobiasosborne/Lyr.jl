## 3. Metric Abstraction

`MetricSpace{D}` (metric.jl:20) is the abstract base. Concrete subtypes must implement:

- `metric(m, x::SVector{4}) -> SMat4d` — covariant g_μν; must accept ForwardDiff Dual numbers.
- `metric_inverse(m, x::SVector{4}) -> SMat4d` — contravariant g^μν.
- `is_singular(m, x::SVec4d) -> Bool` — termination guard.
- `coordinate_bounds(m)` — NamedTuple of valid ranges.

**Optional override** (default: ForwardDiff Jacobian of the 16-component flattened inverse metric):
- `metric_inverse_partials(m, x) -> NTuple{4, SMat4d}` — ∂g^αβ/∂x^μ. All four concrete metrics provide analytic overrides.

**Christoffel symbols** (`christoffel`): analytic for `Schwarzschild{SchwarzschildCoordinates}` (schwarzschild.jl:166, 9 non-zero components listed in docstring) and `Kerr{BoyerLindquist}` (kerr.jl:265, derived from analytic metric partials via the standard Γ formula). Minkowski and SchwarzschildKS do not define `christoffel`; they fall back to the Hamiltonian formulation via `metric_inverse_partials`.

**Per-metric summary**:

| Metric | Coords | `metric_inverse_partials` | `christoffel` | Notes |
|--------|--------|--------------------------|---------------|-------|
| `Minkowski` | Cartesian | ForwardDiff (trivially zero) | — | Test baseline |
| `Schwarzschild{BL}` | (t,r,θ,φ) | Analytic; diagonal | Analytic (9 terms) | Singular at r≤2M+ε |
| `SchwarzschildKS` | (t,x,y,z) | Analytic via Sherman-Morrison l·l structure | — | No pole singularity |
| `Kerr{BL}` | (t,r,θ,φ) | Analytic; 5 components | Analytic (from metric partials) | g_tφ cross-term handled |
| `WeakField` | (t,x,y,z) | Errors (Phase 2 stub) | — | Placeholder only |

## 4. Geodesic Integrator

**State vector** (types.jl:48): `GeodesicState` holds `x::SVec4d` (spacetime coords) and `p::SVec4d` (covariant momentum p_μ). The Hamiltonian is H = ½ g^μν p_μ p_ν; null geodesics have H = 0.

**Equations of motion** (metric.jl:105): `hamiltonian_rhs` returns (dx/dλ, dp/dλ) = (g^μν p_ν, −½ (∂g^αβ/∂x^μ) p_α p_β).

**Steppers** (integrator.jl):
- `RK4` (default): classic 4-stage, 4th-order. `rk4_step` at line 102.
- `Verlet`: Störmer-Verlet (leapfrog), 2nd-order symplectic. `verlet_step` at line 137. Includes polar θ reflection for BL coordinates (θ < 0 or θ > π).

**Adaptive step** (integrator.jl:81): quadratic scale = clamp(((r − 2M)/(6M))², 0.05, 1.0) × dl_base. Near photon sphere r=3M → scale ≈ 0.028 → clamped to 0.05×; at r≥8M → 1.0×.

**Null renormalization** (integrator.jl:187): `renormalize_null` re-projects p_μ onto the null cone every `renorm_interval` steps (default 50 for RK4, 10 for Verlet). Fast diagonal path for Schwarzschild; general quadratic solver A·p_t² + B·p_t + C = 0 for non-diagonal metrics.

**Termination conditions** (integrator.jl:280–315):
- `HORIZON`: r ≤ r_horizon × r_min_factor (default 1.01)
- `ESCAPED`: r ≥ r_max (default 200)
- `SINGULARITY`: `is_singular` returns true
- `HAMILTONIAN_DRIFT`: |H| > h_tolerance (default 1e-6), checked at renorm_interval
- `MAX_STEPS`: step budget (default 10 000) exhausted
- `DISK_HIT`: equatorial plane crossing in disk bounds (render.jl inner loop only; not in `integrate_geodesic`)

Note: `DISK_HIT` is not triggered inside `integrate_geodesic` — disk crossings are detected inline in `render.jl`'s `_trace_pixel_thin_with_p0` to avoid allocating `GeodesicTrace` per pixel.

## 5. Camera and Ray Generation

`GRCamera{M}` (camera.jl:20) stores: metric, position (SVec4d), four_velocity (contravariant u^μ), tetrad (SMat4d, column-major: column a = leg e_a^μ), fov (degrees), resolution.

**Tetrad construction** — `static_observer_tetrad`:
- **Schwarzschild BL** (camera.jl:42): analytic diagonal legs. e_0 = (1/√f, 0,0,0); e_1 = (0,√f,0,0); e_2 = (0,0,1/r,0); e_3 = (0,0,0,1/(r sinθ)).
- **Kerr BL** (camera.jl:81): g_tφ cross-term handled in e_3. e_3^φ = 1/√(−Δ sin²θ/g_tt); e_3^t = −(g_tφ/g_tt)·e_3^φ. Valid only outside the ergosphere.
- **SchwarzschildKS** (schwarzschild_ks.jl:151): spherical basis vectors (ê_r, ê_θ, ê_φ) expressed in Cartesian components, then full Gram-Schmidt orthonormalization against the KS metric to correct for off-diagonal g_ti terms. Pole fallback (ρ < 1e-10·r) uses x and y directions.

**Ray generation** — `pixel_to_momentum` (camera.jl:145): maps pixel (i,j) → screen-space direction (nx, ny, nz) in local frame (forward=e_1, up=e_2, right=e_3), constructs contravariant k^μ = u^μ + nx·e_1^μ + ny·e_2^μ + nz·e_3^μ, then lowers to covariant p_μ = g_μν k^ν. Optional sub-pixel offsets dx,dy for supersampling. The negative `step_size` in `IntegratorConfig` drives backward tracing.

## 6. Matter Models

**ThinDisk** (matter.jl:25): equatorial plane (θ = π/2 in BL; z = 0 in KS). Fields: inner/outer radius, r_isco, T_inner. Emissivity: power-law `(r_in/r)^3` (disk_emissivity). Disk crossing detected via sign change of (θ − π/2) or z between consecutive steps. Novikov-Thorne flux and temperature profiles also provided (`novikov_thorne_flux`, `disk_temperature_nt`).

**ThickDisk** (volumetric.jl:49): analytic density ρ(r,θ) = A·(r_in/r)²·exp(−z²/(2h²)) with h = h_over_r·r. Gaussian vertical profile, power-law radial. Used as `density_source` inside `VolumetricMatter`.

**VolumetricMatter** (volumetric.jl:22): parametric bridge `VolumetricMatter{M,D}`. Wraps any `density_source` D supporting `evaluate_density(d, r, θ, φ)`. Emission: j = ρ²√T (bremsstrahlung-inspired). Absorption: α = 0.34·ρ (electron scattering κ_es). Temperature from `disk_temperature` (Shakura-Sunyaev (r_in/r)^0.75) for emission magnitude; `disk_temperature_nt` for Planck color (Kelvin). Proper step length: √|g_ij dx^i dx^j| from the local metric.

**CelestialSphere** (matter.jl:186): lat-lon texture with bilinear interpolation. Fallback: `checkerboard_sphere` procedural pattern.

## 7. Redshift and Planck Color Pipeline

**Invariant redshift** (redshift.jl:14): 1+z = (p_μ u^μ)_emit / (p_μ u^μ)_obs via `dot(p, u)`. Handles gravitational + Doppler simultaneously. `volumetric_redshift` fetches the local Keplerian `u_emit` at each step; dispatched on Schwarzschild, SchwarzschildKS, and Kerr BL. Safe guard: returns 1.0 within r ≤ 3M to avoid unstable orbits.

**Bolometric scaling** (redshift.jl:80): `doppler_color` applies (1+z)^−4 scaling (broadband bolometric), clamped to [0, 5] to prevent blowup.

**Planck → sRGB pipeline** (redshift.jl):
1. `planck_spectral_radiance(λ, T)` — Planck's law in SI (CODATA 2018 constants).
2. `planck_to_xyz(T)` — trapezoidal integration over CIE 1931 2° CMFs at 5nm (380–780nm, 81 samples).
3. `xyz_to_srgb(X,Y,Z)` — IEC 61966-2-1 3×3 D65 matrix.
4. `srgb_gamma(c)` — IEC 61966-2-1 piecewise gamma.
5. `planck_to_rgb(T)` — full pipeline, normalizes by Y=1 (constant-luminance).
6. `planck_to_rgb_fast(T)` — 2048-entry log-spaced LUT (500K–100000K), linear interpolation. Max error < 0.005/channel. Used in volumetric inner loop.

## 8. Render.jl — Per-Pixel Orchestration

`gr_render_image` (render.jl:325) is the unified entry point. It returns a `height × width` Matrix of RGB NTuple{3,Float64}.

**Dispatch**: if `volume` is provided it takes precedence over `disk`; both may be `nothing`.

**Threading**: `Threads.@threads :dynamic` over image rows when `config.use_threads` (default true). No per-ray allocation; all state is stack-allocated via `SVec4d`/`SMat4d`.

**Supersampling**: stratified N×N grid, `spp` samples per pixel (default 1). `pixel_to_momentum` accepts sub-pixel offsets.

**Thin-disk inner loop** (`_trace_pixel_thin_with_p0`, render.jl:91): inline `_check_disk_crossing_bare` avoids `GeodesicState` allocation. On disk hit: computes emissivity, optionally applies (1+z)^−4 redshift, returns `blackbody_color` (simple warm ramp, not full Planck).

**Volumetric inner loop** (`_trace_pixel_with_p0`, render.jl:170): accumulates (R,G,B) emission and optical depth τ at each step. Early exit at τ > 8. Uses `planck_to_rgb_fast` for Planck color at observed temperature T_obs = T_emit/(1+z). Final compositing in `_volumetric_final_color` (render.jl:274): transmittance·bg + accumulated; **Reinhard tone-mapping** r/(1+r) per channel (see open issues below).

**Pole guard** (render.jl:202): volumetric loop terminates and blends accumulated color with sky if θ < 1e-3 or θ > π−1e-3 (BL only; KS has no pole issue).

**GPU**: No GPU GR hooks are present in this module. The memory notes mention GPU GR (issue `kzfy`) was closed; the current codebase is CPU-only. GPU delta-tracking for the flat VDB renderer is separate (KernelAbstractions.jl in `src/`).

## 9. Open Issues Visible in Source

A grep for `TODO`, `FIXME`, `XXX`, `sorry`, `@warn` across all `src/GR/` files returns **no matches**. Issues are instead documented externally. From the source structure and comments, the following are observable:

1. **`EddingtonFinkelstein` coordinates unimplemented** (schwarzschild.jl:17): struct declared as "Horizon-penetrating (Phase 2)" but no `metric`/`metric_inverse` methods exist. Any use will produce a `MethodError`.

2. **`KerrSchild` coordinates unimplemented** (kerr.jl:23): same situation — struct declared but no methods. Phase 2 placeholder.

3. **`WeakField` metric errors on use** (stubs/weak_field.jl:18): `metric` and `metric_inverse` call `error(...)` explicitly. Type is exported and constructible but non-functional.

4. **Reinhard tone-mapping washes out volumetric colors** (render.jl:281): the per-channel `r/(1+r)` application in `_volumetric_final_color` is acknowledged in project notes as causing washed-out appearance. No in-source marker but the code comment says "Reinhard tone mapping for HDR → [0,1]".

5. **Pole artifact in BL volumetric rendering** (render.jl:202): the early-exit pole guard at θ < 1e-3 is a mitigation, not a fix — rays near the pole that accumulate emission before hitting the guard will be composited inconsistently.

6. **Thin-disk uses `blackbody_color` (simple ramp), not `planck_to_rgb`** (render.jl:127): the thin-disk path returns `blackbody_color(clamp(intensity * 5.0, 0.0, 2.0))`, a simple ad-hoc ramp, while volumetric uses the full CIE Planck pipeline. This is an inconsistency rather than a bug.

7. **Adaptive step uses M extracted as `rh/2`** (integrator.jl:272): `M_val = rh / 2.0` works for Schwarzschild (r_h = 2M) but is physically incorrect for Kerr (r_h = M + √(M²−a²) ≠ 2M). The adaptive scaling will therefore use a wrong effective mass for Kerr black holes.

8. **`integrate_geodesic` never sets `DISK_HIT`** (integrator.jl): the `DISK_HIT` termination reason is defined but unreachable via `integrate_geodesic` — disk detection is done only in the render loop's inline version. Any user calling `integrate_geodesic` directly for disk tracing will never see this reason.

9. **RK4 upgrade (`esr2`) open**: project notes indicate an open issue for upgrading to a higher-order adaptive stepper; the current RK4 is non-symplectic and fixed-order.
